# FunSearch LLM 生成策略问题修复计划

## 问题分析

### 核心问题
- LLM 只生成均值策略，导致每次 island 的值都相同
- 每轮进化无效，没有产生策略多样性
- 不确定 LLM 是否正确接收到前一轮的结果

### 技术原因
1. **提示词问题**：当前 LLM 提示词固定，没有包含前一轮的最佳策略信息
2. **代码修复逻辑**：过度的代码修复可能将多样化策略强制转换为简单均值策略
3. **API 调用**：可能需要优化 API 调用参数以提高生成多样性
4. **数据库集成**：LLM 没有从 ProgramsDatabase 中获取前一轮的最佳结果

## 解决方案

### 1. 优化 LLM 提示词
- 修改 `sampler.py` 中的提示词，包含前一轮的最佳策略信息
- 添加策略性能指标到提示词中，引导 LLM 生成更优策略
- 增加策略多样性要求，明确禁止简单均值策略

### 2. 改进代码修复逻辑
- 减少过度的代码修复，保留 LLM 生成的多样化策略
- 只修复严重的语法错误和逻辑错误
- 移除强制替换为均值策略的逻辑

### 3. 优化 API 调用参数
- 调整 temperature 参数，提高生成多样性
- 增加 max_tokens，允许更复杂的策略生成
- 选择更适合代码生成的模型

### 4. 集成 ProgramsDatabase 信息
- 修改 `Sampler` 类，在生成提示时包含前一轮的最佳策略
- 确保 LLM 能够获取到历史最佳策略的代码和性能指标
- 实现策略进化的连续性

## 实施步骤

### 步骤 1：分析当前 LLM 行为
- 运行现有代码，观察 LLM 生成的策略
- 检查 ProgramsDatabase 中的最佳策略存储
- 验证 API 调用是否正常

### 步骤 2：修改 sampler.py
- 更新 LLM 类的 `_draw_sample` 方法
- 优化提示词结构，包含历史最佳策略
- 调整代码修复逻辑，保留策略多样性

### 步骤 3：集成历史信息
- 修改 `Sampler.sample()` 方法，从数据库获取历史最佳策略
- 将历史策略信息传递给 LLM
- 确保每轮进化都基于前一轮的最佳结果

### 步骤 4：优化 API 参数
- 测试不同的 temperature 值（0.7-1.0）
- 调整 max_tokens 以允许更复杂的策略
- 验证 API 调用的稳定性

### 步骤 5：测试和验证
- 运行完整的 FunSearch 实验
- 观察策略多样性和性能改进
- 验证每轮进化是否有效

## 技术要点

### API 文档参考
- 使用 `https://api.bltcy.ai/` 的 API 接口
- 参考 `https://gpt-best.apifox.cn/llms.txt` 文档
- 确保使用正确的模型和参数

### 策略多样性
- 鼓励 LLM 生成不同类型的策略（趋势跟踪、均值回归、波动率加权等）
- 提供多样化的策略示例
- 避免生成简单的均值策略

### 性能评估
- 确保生成的策略能够通过评估函数
- 监控每轮进化的性能改进
- 验证策略在不同 market_state 下的表现

## 预期结果

- LLM 能够生成多样化的投资策略
- 每轮进化都基于前一轮的最佳结果
- 不同 island 生成不同的策略
- 策略性能逐步提升
- FunSearch 能够找到优于基准策略的投资策略

## 风险控制

- 确保 API 调用的稳定性和可靠性
- 防止生成无效或有害的策略代码
- 保持代码修复逻辑的平衡，既确保代码正确性又保留策略多样性
- 监控计算资源使用，避免过度消耗
